# カメラで線を追って走る車
## ―― ライントレースカーの仕組み ――

自分でカーブを曲がりながら、床の線の上を走るロボット

<!--
🗣️ 話すこと：
このロボットは、人が運転しなくても、カメラで床に引かれた線を見て、
自分でその上を走り続ける車です。今日はこれが「どうやって線を見つけて」
「どうやって曲がっているのか」を、順番に説明します。
-->

---

# 一言でいうと

> ## カメラで足元の線を見て、
> ## 右にズレたら少し右、左にズレたら少し左へ、
> ## **左右のタイヤの速さを変えて**、
> ## **1秒間に何十回も**微調整しながら走っている。

<!--
🗣️ 話すこと：
最初に結論だけ言っておきます。この車は、足元の線を見て、線が右にズレていたら
少し右、左にズレていたら少し左へ、左右のタイヤの速さを変えて曲がります。
これを1秒間に何十回も高速で繰り返しているだけです。
人が自転車で「ちょっと右にズレたな、じゃあ左」を無意識にやっているのと同じことを、
機械が超高速でやっている、と思ってください。
-->

---

# 全体の流れ（このループを高速で繰り返す）

```
 ① カメラで写真をとる
        ↓
 ② 写真から「線」を見つける
        ↓
 ③ 線が右・左・真ん中どこにあるか測る
        ↓
 ④ ズレに応じて「どれだけ曲がるか」を計算
        ↓
 ⑤ 左右のタイヤの速さを決めてモーターを回す
        ↓
     （①に戻る）
```

<!--
🗣️ 話すこと：
処理はこの5ステップの繰り返しです。写真をとる、線を探す、線の位置を測る、
どれだけ曲がるか計算する、タイヤを回す。これを止まらずグルグル回し続けます。
ここから1ステップずつ見ていきます。
-->

---

# ① カメラで写真をとる

- 頭についた小型カメラ（**OV2640**）で撮影
- **白黒**（明るさだけ）／**160×120ピクセル**の小さな写真
- 色は使わない → **明るい/暗いだけ**で線は判別できる → 処理が速い

### 画面全部は見ない ＝ ROI（注目エリア）

```
┌─────────────┐
│             │ ← 見ない（遠くの景色）
│             │
│░░░░░░░░░░░░░│ ← ここだけ見る（足元の線）ROI
└─────────────┘
```
画面の高さの **65%〜95%** の帯（＝手前の床）だけを処理

<!--
🗣️ 話すこと：
まずカメラで写真をとります。ポイントは2つ。
1つ目、カラーではなく白黒の小さい写真です。ライントレースに必要なのは
「色」ではなく「明るいか暗いか」だけなので、白黒で十分。データが軽くて速いんです。
2つ目、写真全部は見ません。遠くの線を見てもすぐ曲がれないので、
画面の下のほう＝今まさに通過する足元の帯だけを見ます。これをROI（注目エリア）と呼びます。
-->

---

# ② 写真から「線」を見つける

**線 ＝ まわりより暗い（または明るい）ところ**

1. まず足元エリアの**平均の明るさ**を計算（＝床の明るさ）
2. そこから**一定以上暗いピクセル**を「線だ」と判定

### なぜ「平均から」で判定するのか

- 「明るさ100より暗ければ線」と**固定値**にすると、照明が変わると失敗する
- 「**まわりの平均より暗ければ線**」なら、明るい部屋でも暗い部屋でもOK
- 平均は毎回とり直す → **コースや照明が変わっても自動でついていける**

<!--
🗣️ 話すこと：
次に、その足元の写真から線を見つけます。線は、白い床なら黒く、
つまりまわりより暗く写ります。
そこで、まず足元全体の平均の明るさ＝床の明るさを計算して、
そこから一定以上暗いところを「これは線だ」と判定します。
ここが賢いところで、「明るさ100より下なら線」と固定の数字で決めると、
部屋の明るさが変わったとたん失敗します。でも「まわりの平均より暗ければ線」なら、
明るい部屋でも暗い部屋でも同じように線を見つけられる。しかも平均は毎回計算し直すので、
コースや照明が変わっても自動で対応できます。
-->

---

# ③ 線が右・左・真ん中どこにあるか測る

線と判定したピクセルの**重心（＝線の真ん中の位置）**を計算

```
      画面の真ん中
           │
   ┌───────┼───────┐
   │       │  ███  │  線が右にある
   └───────┼───────┘
           │   ↑ 重心はここ →「右にズレている！」
```

**重心と画面中央のズレ ＝「誤差」**

- 線が右 → 誤差プラス｜線が左 → 誤差マイナス｜真ん中 → ゼロ（まっすぐでOK）
- 線ピクセルが少なすぎたら（**8個未満**）→「線を見失った」と判断して**停止**

<!--
🗣️ 話すこと：
線が見つかったら、それが右にあるのか左にあるのかを測ります。
線ピクセルの位置を平均した「重心」＝線の真ん中の位置を出して、
それが画面の中央からどれだけズレているかを計算します。これを「誤差」と呼びます。
右にズレていればプラス、左ならマイナス、真ん中ならゼロ。
なお、線ピクセルが極端に少ないときは「線を見失った」と判断して、
暴走しないよう安全に止まります。
-->

---

# ④ どれだけ曲がるか計算する（ここが頭脳）

### 基本：ズレが大きいほど大きく曲がる ＝ **P制御**

```
曲がる量 ＝ ズレの大きさ × 0.9（KP：調整つまみ）
```

- ちょっと右にズレた → 少し左へ
- 大きく右にズレた → ぐいっと左へ

人が運転で「大きくズレたら大きくハンドルを切る」のと同じ感覚

<!--
🗣️ 話すこと：
誤差、つまりどれだけズレているかが分かったら、どれだけ曲がればいいかを計算します。
一番基本の考え方はシンプルで、「ズレが大きいほど強く曲がる」。これをP制御と言います。
ズレの大きさに0.9という倍率をかけて曲がる量にしています。この0.9は調整つまみで、
大きくすると反応が機敏に、小さくするとおっとりします。
ただしP制御だけだと問題が起きるので、ここに4つの改良を足しています。
-->

---

# ④ 4つの改良で「速く・安定して」走らせる

| 改良 | 問題 | 対策 |
|---|---|---|
| **D制御** | 曲がりすぎて左右にフラフラ（蛇行） | ズレが急に縮んだら曲げるのをゆるめる（先読み） |
| **不感帯** | 直線でノイズに反応してフラつく | 中央のごく近くはズレを無視してまっすぐ |
| **カーブ減速** | 速すぎて急カーブでコースアウト | ズレが大きい＝急カーブと見て自動減速 |
| **傾き補正** | カーブ直後、車体だけ斜めでフラつく | 足元を手前/先の2つに分け傾きを検出し先回り修正 |

P制御 ＋ D制御 ＝ **PD制御**

<!--
🗣️ 話すこと：
1つ目、D制御。P制御だけだと勢いで曲がりすぎて、今度は反対にズレて…と
左右にフラフラする「蛇行」が起きます。そこで「ズレが急に縮んでいるなら
もう曲げるのをゆるめる」という先読みを足して、揺れを抑えます。
2つ目、不感帯。直線ではカメラのノイズで重心が1〜2ピクセル動きます。
これに反応するとフラつくので、中央のごく近くではズレを無視してまっすぐ走ります。
3つ目、カーブ減速。速すぎると急カーブで飛び出すので、ズレが大きい＝急カーブだと
判断したら自動でスピードを落とします。直線では落とさないので速さは損ないません。
4つ目、傾き補正。カーブを抜けた直後、線の真ん中に戻っていても車体だけ斜めのことがあります。
そこで足元を手前と先の2つに分けて、線の写り方の差から車体の傾きを見抜き、
ズレる前に先回りでまっすぐに戻します。
この4つで、速く走らせても安定するようにしています。
-->

# ⑤ 左右のタイヤの速さの差で曲がる

このロボットに**ハンドルは無い**。**戦車と同じ**で左右のタイヤの速さ差で曲がる。

```
左タイヤの速さ ＝ 基本スピード ＋ 曲がる量
右タイヤの速さ ＝ 基本スピード － 曲がる量
```

### 例：線が右にズレているとき（右に曲がりたい）

```
   左タイヤ 速い ━━━▶
                     ＼ 右へ曲がる
   右タイヤ 遅い  ━▶ ／
```
外側（左）を速く・内側（右）を遅く → 遅いほうの右へ曲がる
※線が真ん中なら左右同速 ＝ まっすぐ

<!--
🗣️ 話すこと：
計算した「曲がる量」を、実際の動きに変えます。
この車にはハンドルがありません。戦車やキャタピラ車と同じで、
左右のタイヤの速さを変えて曲がります。
基本スピードに曲がる量を、左は足す・右は引く。すると、たとえば右に曲がりたいときは、
外側の左タイヤが速く、内側の右タイヤが遅くなって、遅いほうの右へ曲がっていきます。
線が真ん中なら左右同じ速さで、まっすぐ進みます。
-->

---

# ⑤ モーターを回す信号（ESCの話）

速さ（%）を、モーターが分かる**信号（パルスの長さ）**に翻訳して送る（ラジコンと同じ）

- **1500マイクロ秒 → 停止**（真ん中）
- それより**長い → 前進**／**短い → 後進**

### 現実に合わせた細かい補正

- **動き出しの底上げ**：弱い信号だと摩擦で全く動かない → 最低限動く所まで底上げ
- **左右の個体差補正**：同じ信号でも左右で速さが違う → 倍率と中立値で調整
- **回転方向の反転**：タイヤが逆向きなら信号を反転（`INVERT_L/R`）

<!--
🗣️ 話すこと：
最後に、計算した速さをモーターに伝えます。このモーターはESC、
電子スピードコントローラーという、ラジコンと同じ部品で動いていて、
「パルスの長さ」で速さを伝えます。1500マイクロ秒が停止、
それより長ければ前進、短ければ後進です。
現実のモーターは教科書通りにいかないので、弱い信号だと摩擦で動かない分の底上げや、
左右の個体差の補正、タイヤの向きの反転など、細かい調整が入っています。
-->

# 動作モードと安全のしくみ

| キー | モード | 何をする？ |
|---|---|---|
| `l` | LINE | カメラを見て**自分で線をたどって走る**（本命） |
| `r` | REMOTE | PCから**ラジコンのように**操作 |
| `i` | IDLE | **タイヤは止めたまま**、線を見つけているか画面で確認だけ |

- **オートスタート**：PCなしでバッテリー電源を入れると**3秒後に自動走行**
- **フェイルセーフ**：リモコン中は通信が**0.5秒**途切れたら自動停止
- **ライン消失で停止**：線を見失ったら止まる（コースアウト防止）

<!--
🗣️ 話すこと：
モードは3つ。lで自分で線をたどる自律走行、rでPCからのラジコン操作、
そしてiが便利で、タイヤを止めたまま「ちゃんと線を見つけられているか」を
画面の文字だけで確認できます。いきなり走らせず、まず目のテストができるわけです。
安全面では、バッテリーだけで電源を入れると3秒後に自動で走り出す本番モード、
リモコン中に通信が0.5秒切れたら止まるフェイルセーフ、
線を見失ったら止まる仕組みが入っています。
-->

# システム全体の構成（通信の流れ）

```
[Mac / Processing] ──USBシリアル──▶ [中継ESP32] ──ESP-NOW無線──▶ [走行機 ESP32-S3]
   %を入力・可視化                    シリアル→無線に変換           カメラで線を検出
                                                                    ↓ ESC信号
                                                              [左右モーター]
```

- 走行機（**XIAO ESP32-S3 Sense**）がカメラで**自律走行**するのが主役
- MacはESP-NOWを直接話せない → **USB接続の中継ESP32**が無線に変換
- 無線は**ESP-NOW**（ペアリング不要・つなぎ直し不要で復帰が速い）

<!--
🗣️ 話すこと：
最後に全体の構成です。主役は、カメラを載せてカメラで走る走行機、
XIAO ESP32-S3というマイコンです。基本はこれ単体で自律走行します。
リモコンで動かしたいときは、MacのProcessingというソフトから指令を出しますが、
MacはESP-NOWという無線を直接話せないので、USBでつないだ中継用のマイコンが
シリアル通信を無線に変換して走行機に飛ばします。
この無線は接続の手続きが要らないので、電波が一瞬切れても次の指令ですぐ復帰できます。
-->

# まとめ

この車を支えているのは、シンプルな4つの仕組み：

1. **白黒画像から線を見つける**（まわりの平均より暗い/明るいで判定）
2. **線のズレを測る**（重心と誤差）
3. **ズレから曲がる量を計算する**（PD制御＋4つの改良）
4. **左右タイヤの速さの差で曲がる**（戦車と同じ原理）

> ### カメラで足元の線を見て、左右のタイヤの速さを変えながら、
> ### 1秒に何十回も微調整して走っている。

この4つの仕組みでできています。
